About

This website is delivered using an Application Programming Interface (API), which means its content is machine-readable as well as human-readable. It re-delivers the publicly available content of directory.gov.au according to Semantic Web principles.

The goal of establishing this website is to demonstrate how Semantic Web versions of the directory.gov.au content could look in the future and how Linked Data systems may interact with it.

Many of the things this website demonstrates rely on the Australian Government Linked Data Working Group (AGLDWG) and its management of persistent Internet identifiers for Linked Data for Australian government.

How the data is produced

The data for this API is taken from directory.gov.au's XML data feed made available in the data.gov.au catalogue and then converted into the Semantic Web format RDF using this code. That RDF data is then housed in an Apache Jena graph database made available on the web via the Fuseki UI which this API then queries as people or machines request particular web addresses.

The code for this API is available from CSIRO Land & Water's Environmental Informatics Group's GitHub account as is the Python code for the underling mechanics to get Linked Data on the web.

Status of the tool

This API is only a small demonstrator. It will hopefully be replaced within the 2018/2019 Financial Year by a more robust tool that delivers RDF data for directory.gov.au data perhaps incorporated within directory.gov.au itself. The work to built this more robust tool is now being discussed by the Department of Finance (owners of directory.gov.au) and the CSIRO.

Persistent Identifier Demonstration

This API demonstrates persistent identifiers (PIDs) in action but, since it's a demo, it includes the word test in all PIDs that would be removed for a production system.

PID web addresses used to make requests for content from this API are long-term stable and can thus can be quoted and used without fear that they will become broken links.

For Geoscience Australia (GA):

  • Instead of https://www.directory.gov.au/portfolios/industry-innovation-and-science-part-jobs-and-innovation-portfolio/geoscience-australia - the human-readable, directory.gov.au address
  • or https://www.ga.gov.au - GA's own website addresses

PID Type examples

The various PIDs used by this website API are:

  • a Linked Data dataset PID - http://test.linked.data.gov.au/dataset/auorg that points to the main page of this API
  • an ontology - http://test.linked.data.gov.au/def/auorg which defines the custom thing in this dataset
  • a series of register PIDs - for the main classes of things in this API, e.g. /org/ for organisations
  • an instance PID - i.e. the web adress for one particular things in this dataset, e.g. http://test.linked.data.gov.au/org/O-000887

Machine-readability

In addition to simple web pages for each thing in this API, machine-readable forms of all content are available.

You can get to the machine-readable version of things by following a simple version of the Linked Data Platform specification:

  • add a _format query string to any URL to get a machine-readable format
    • e.g. Geoscience Australia in JSON-LD: http://test.linked.data.gov.au/org/O-000887?_format=application/rdf+json
  • add a _view query string to any URL to get a supported model view
    • e.g. Geoscience Australia according to this dataset's own AU ORG Ontology (which is default): http://test.linked.data.gov.au/org/O-000887?_view=auorg

See the Alternates Views of any Linked Data object in the API to see all the views & formats available for it by adding _view=alternates, e.g. for Geoscience Australia:

  • http://test.linked.data.gov.au/org/O-000887?_view=alternates

Registers

Top-level registers

This API delivers 4 top-level Registers which are:

  • board/ - a list of auorg:Boards
  • org/ - a list of org:Organizations
  • person/ - a list of foaf:Persons
  • portfolio/ - a list of auorg:Portfolios

These 4 top-level registers have been assigned PIDs that are 'top-level' in terms of their placement within linked.data.gov.au which is due to their recognition by the AGLDWG. The AGLDWG can share top-level reigsters between multiple datasets so that /org/ could draw instances of org:Organization from this API and others. For the moment it doesn't.

Secondary Registers

Other, less significant, registers of things delivered by this API that have not yet been recognised by the AGLDWG would be delivered at .../dataset/auorg/{REGISTER-NAME}/. For example, the director.gov.au 'non-Board' objects would be at .../dataset/auorg/nonboard/.

In test mode, these secondary registers are not available.

Register of Registers

In addition to the 4 main registers that this API delivers, it also has a top-level register that lists all other registers:

  • reg/

Automated clients, such as Linked Data Web Crawlers, can use this Register of Registers to discover all this API's content.
